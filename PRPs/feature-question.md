# 需求澄清记录

## 第一次需求澄清

### 1. 用户角色权限设计
**问题**: 系统需要支持哪些用户角色？不同角色的权限划分是什么？

**回答**:
- 系统支持管理员、普通用户两种角色
- 管理员只有一名，通过配置表配置，拥有整个系统的控制权
- 只有管理员可以创建和修改文章
- 普通用户需要在系统中进行注册，只能对文章进行评论和点赞收藏等操作，不可以创建文章

### 2. 部署架构
**问题**: 目标部署环境、数据库选择、是否支持分布式部署？

**回答**:
- 选择云服务器
- 数据库使用SQLite
- 单机部署

### 3. 随念模块特性
**问题**: 随念与博客文章的区别、内容格式、多媒体支持、展示方式？

**回答**:
- 内容限制在300字以内
- 纯文本格式
- 支持图片、视频，复用图床模块
- 展示方式采用时间线方式
- 和博客文章采用相同的分类和标签系统，但通过类型字段分割开
- 在搜索时不同时展示

### 4. GitHub同步功能
**问题**: 同步内容范围、同步方向、触发方式、仓库结构？

**回答**:
- 采用markdown方式同步，随念也需要同步，图片等媒体文件不需要同步
- 单向同步，系统 -> Github
- 自动定时同步和手动触发同步和文章发布/更新时自动同步都需要
- GitHub仓库的结构按分类组织

### 5. 系统规模预期
**问题**: 用户规模、内容规模、访问量预期？

**回答**:
- 预期同时在线用户数量为100人以内
- 注册用户总数在10000以内
- 预期文章数量在1000篇以内
- 随念内容预期每天两篇，数量在1000以内
- 预期日均访问量为100人次
- 没有高并发场景

### 6. 用户注册登录扩展性
**问题**: 用户注册登录是否需要支持多种方式？

**回答**:
- 需要支持后期拓展多种登录方式：GitHub、微信、Google、手机号、邮箱
- 当前只实现邮箱验证码登录
- 要求设计良好的扩展架构，便于后期添加新的登录方式
- 支持多种登录方式绑定到同一用户账号
